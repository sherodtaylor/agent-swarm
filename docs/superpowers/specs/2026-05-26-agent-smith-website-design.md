# agent-smith website — design spec

**Status:** approved 2026-05-26 (Sherod, via Matrix brainstorm);
README-sync revision 2026-05-27
**Owner:** DevBot
**Last updated:** 2026-05-27

A public-facing website for agent-smith — your engineering team,
running in Kubernetes — hosted on GitHub Pages. Serves adopters (who
might self-host), contributors (who might open PRs), and curious
onlookers (who want to understand what this is) — equally.

The site is multi-interface in its framing from day one, matching the
README: agents are reachable from a Matrix room, from a phone, or via
the Claude desktop app — the interface is up to the operator, the
engineering capability is always there.

> **Canonical-wording convention:** the README is the single source
> of truth for product positioning (tagline, hero sub, the "What your
> team can do" bullets, the production-grade substrate paragraph).
> Whenever the README moves, this spec follows in a small sync PR —
> not the other way around. References below quote the current
> README on `main` as of the *Last updated* date above.

## 1. Information architecture

A single site with three top-level surfaces. URL prefix is
`/agent-smith/` while hosted on `sherodtaylor.github.io/agent-smith/`;
flips to `/` cleanly when a custom domain is pointed (see §5).

### 1.1 `/` — landing (single long-scroll page)

Six sections, in order:

1. **Hero.** Faux-tmux session, two panes:
   - Desktop: panes side-by-side. Left pane = command prompt + ASCII
     logo + tagline + sub + dual CTA. Right pane = live `$ tail -f
     #audit` style scroll of the 5 most recent log entries (sourced
     from the same content collection as `/log`).
   - Mobile: panes stack vertically. Top = headline pane. Bottom =
     compact terminal card showing the 3 most recent log entries.
   - Top edge of the hero is a tmux-style status bar showing
     `[agent-smith] [0:main] ● 2 agents · PRs this week: 11 · last
     release: v0.1.16`. All values sourced from gh-api at build time
     (no cluster reach-in — see §3.3); refresh cadence = each deploy.
2. **What this is.** 3-paragraph narrative. Lead line mirrors the
   README opening: "agent-smith deploys Claude Code as persistent,
   long-lived engineering agents inside Kubernetes pods. Each agent
   has a permanent workspace with real cluster credentials, follows
   the same git workflow as a human teammate, and works
   autonomously until the task is done — feature branches,
   conventional commits, pull requests, review comments addressed,
   merged." Editorial layout — oversized lead line, restrained body.
3. **What your team can do.** 5-bullet list mirroring the README
   section of the same name (single column on mobile, terminal-styled
   bullets):
   - **Owns a persistent workspace** — full filesystem + shell access
     on a long-lived volume with real cluster credentials.
   - **Follows the full engineering workflow** — reads, writes,
     opens the PR, addresses review, merges. The whole loop.
   - **Watches its own PRs** — a `Stop`-hook reruns the agent when
     unaddressed review comments appear.
   - **Coordinates with teammates** — one agent opens a PR, the
     other reviews it end-to-end and posts inline findings. NATS is
     the durable audit log.
   - **Never holds production secrets** — stub tokens are swapped
     for real credentials at the network boundary by an egress
     firewall.

   Sub-line after the bullets: "Reach them from a Matrix room, from
   your phone, or via the Claude desktop app. The interface is up
   to you; the engineering capability is always there." (verbatim
   from README)
4. **Under the hood.** Architecture summary using ASCII-box diagrams,
   not SVGs. Covers: StatefulSet per agent, init container assembling
   `~/.claude/`, tmux dance (single `claude` process with channels +
   `--remote-control`), iron-proxy egress credential firewall, GitOps
   via Flux, secrets sourced from Infisical via ExternalSecrets,
   observability via VictoriaMetrics / VictoriaLogs. Each subsection
   links to the deeper docs page. The closing line lifts the README's
   own: "These agents ship work that ends up in `main`."
5. **The crew right now.** Build-time current status block (no
   prose; refreshed each deploy, not real-time). Per-agent: name,
   role, last shipped PR, last activity timestamp. Sourced from
   `website/src/data/crew-status.json`, regenerated by the deploy
   workflow from gh-api data.
6. **Footer.** Links: repo · license · contributing · log · docs.

### 1.2 `/docs` — Starlight

Astro Starlight under the same shared visual chrome. Section list:

- Getting Started
- Architecture
- Agents (persona format, adding a new agent)
- Security (iron-proxy)
- Operations (release, oauth-401, agent-down, ci-failure,
  secret-rotation — pulled from the existing `docs/runbooks/`)
- Contributing
- Roadmap (renders `docs/roadmap-v1.md` as-is — single source of
  truth)

Starlight's default light/dark toggle is removed; site is dark-only
for v1.

### 1.3 `/log` — chronological event feed

Reverse-time list of notable runs and releases. Each entry one line
plus an optional expand for body/links. See §4.3 for entry format and
data source.

## 2. Visual system

Dark mode only for v1. Terminal-as-zine register: mono everywhere,
editorial layout, asymmetric grid on wide viewports, generous
whitespace.

### 2.1 Palette

| Token | Hex | Use |
|---|---|---|
| `--bg` | `#0b0d10` | page background (near-black, slight blue tint — unfocused tmux pane, not pure void) |
| `--bg-elev` | `#13171b` | panes, cards, code blocks, log-entry surfaces |
| `--fg` | `#d4d7dc` | body text (warm off-white, not pure white) |
| `--fg-muted` | `#7a818c` | timestamps, meta, ASCII box edges, run-ids |
| `--accent` | `#5fbf8d` | prompts (`$`), live cursor, "ok" status, hero CTAs |
| `--accent-warn` | `#d4a85f` | "warn", in-progress, log entries needing attention |
| `--accent-err` | `#cf6679` | sparingly: fail, blocked, hero "danger" badges |

Discipline: one accent at a time per page region. Never three loud
accents at once.

Contrast check (WCAG): `--fg` on `--bg` = 13.4:1 (AAA); `--accent` on
`--bg` = 6.7:1 (AAA for body, AA for large text). All passing.

### 2.2 Typography

All monospace. Three roles, one family (or two if Berkeley Mono is
licensed for display):

| Role | Family | Weight | Size |
|---|---|---|---|
| Display (hero, section headers) | Berkeley Mono if licensed, else JetBrains Mono | 700 | `clamp(2rem, 5vw, 4.5rem)`, tight tracking |
| Body (prose, lists) | JetBrains Mono | 400 | `15px / 1.65 line-height`, `max-width: 66ch` |
| Code / data (status, log entries, ASCII) | JetBrains Mono | 400 | `13px / 1.5`, denser |

Self-hosted via `@fontsource/jetbrains-mono` (no Google Fonts,
`font-display: swap`).

### 2.3 Spacing

8px base scale: `4 / 8 / 12 / 16 / 24 / 32 / 48 / 80 / 128`.
Generous in editorial sections (80+ between blocks), tight in
terminal-styled regions (8–16, mimicking real tmux panes).

### 2.4 ASCII box library

Reusable conventions for diagrams. All ASCII renders go through a
shared `.ascii` CSS class: `<pre>`, mono, `--fg-muted` for edges,
`--fg` for labels, `--accent` for live or highlighted values.

```
┌─ devbot ──┐    flow arrows: ──>   pipes: │
│  status   │    panes: light box-drawing (U+2500 family)
└───────────┘    scope boundaries: ╔══╗ heavy
```

Every ASCII diagram has `role="img"` and an `aria-label` describing
the diagram in prose (see §5.5).

### 2.5 Live elements

- Block cursor `▎` in `--accent`, CSS `@keyframes` blink (0.5s on /
  0.5s off). Used in the hero prompt and `[currently running]`
  status indicators. Honors `prefers-reduced-motion` (static cursor
  fallback).
- Status glyphs: `●` running, `✓` ok, `!` warn, `✗` fail — in the
  appropriate accent color. No SVG icon set.

### 2.6 Mobile breakpoints

| Range | Behaviour |
|---|---|
| `< 640px` | Single column. Hero panes stack vertically. Headlines drop to bottom of `clamp()`. ASCII diagrams in horizontally-scrollable `<pre>` (acceptable: this is a terminal site). |
| `640–1024px` | Tablet. Hero panes side-by-side at 60/40 split. |
| `> 1024px` | Desktop. Hero panes 50/50. Editorial asymmetric grid for content sections. |

## 3. Tech stack

### 3.1 Framework

**Astro 5 + Starlight.** Astro for the landing and `/log` (MDX +
island components for the live status block, minimal JS shipped).
Starlight for `/docs` (nav + search + dark chrome out of the box,
respects the token system).

Rejected alternatives:
- **11ty / plain HTML** — fine for landing, painful for docs nav +
  search; would need hand-rolled equivalents.
- **Next.js static export** — heavier runtime, more config, no
  built-in docs story.
- **Hugo** — fast and lovely, but JS islands for the live status
  block are awkward.

### 3.2 Repo layout

Site lives in this repo under `website/` (not a separate repo), so the
site is always versioned with the code it documents.

```
website/
├── astro.config.mjs
├── package.json
├── tsconfig.json
├── README.md                    # local dev + ADR for custom domain swap
├── src/
│   ├── pages/
│   │   ├── index.astro          # landing
│   │   ├── 404.astro
│   │   └── log/index.astro      # log feed
│   ├── content/
│   │   ├── config.ts            # collection schemas
│   │   ├── docs/                # Starlight content
│   │   └── log/                 # YYYY-MM-DD-<slug>.mdx entries
│   ├── components/
│   │   ├── TmuxStatusBar.astro
│   │   ├── HeroPane.astro
│   │   ├── AuditTail.astro
│   │   ├── AsciiBox.astro
│   │   ├── StatusBadge.astro
│   │   └── LottiePlayer.astro   # lazy-load wrapper
│   ├── styles/
│   │   ├── tokens.css           # palette, type, spacing
│   │   └── global.css
│   ├── animations/              # Lottie JSON sources (3 only — see 3.4)
│   └── data/
│       └── crew-status.json     # generated each build by website.yml
```

### 3.3 Build / deploy

New workflow `.github/workflows/website.yml`:

- Triggers: push to `main` touching `website/**`, `docs/**`, or
  `website/src/content/log/**`. Also on `workflow_dispatch`.
- Steps: checkout → `oven/bun:1` → `cd website && bun install --frozen-lockfile` → run `website/scripts/refresh-crew-status.ts` (small Bun script that calls gh-api, no cluster reach-in for v1, writes `src/data/crew-status.json`) → `bun run build` → `actions/upload-pages-artifact` → `actions/deploy-pages`.
- Concurrency group `pages` so a slow build can't deploy stale output over a newer one.

GitHub Pages source: "GitHub Actions" (not branch-based).

Astro config:
- `site: 'https://sherodtaylor.github.io'`
- `base: '/agent-smith'`
- `output: 'static'`
- `trailingSlash: 'never'`

### 3.4 Runtime dependencies

| Dep | Purpose |
|---|---|
| `astro` | framework |
| `@astrojs/starlight` | docs section |
| `@astrojs/mdx` | log entries authored as MDX |
| `@astrojs/sitemap` | sitemap.xml |
| `@fontsource/jetbrains-mono` | self-hosted body font (no Google Fonts request) |
| `@lottiefiles/lottie-player` | Lottie web component, lazy-loaded |

No CSS framework. No icon library. No analytics SDK. No CDN runtime
dependency.

### 3.5 Lottie animations (scoped, sparse)

Lottie is scoped to three moments only — anywhere else risks fighting
the mono/ASCII register.

| # | Moment | Behaviour |
|---|---|---|
| 1 | Hero boot sequence | 2–3s opening that simulates the agent booting: `$ agent-smith --hello` types in, panes resolve. Loops static after first play. |
| 2 | Architecture section transition | When scrolled into view, ASCII boxes animate in line-by-line (as if a terminal is rendering them). Plays once on scroll-into-view. |
| 3 | Log section newest-entry highlight | Newest entry pulses softly (accent-color fade) on landing. Cheap to drop if it feels gimmicky after first deploy. |

Discipline:
- Strict `prefers-reduced-motion` honoring — every Lottie has a
  static fallback frame.
- No animation longer than 3 seconds. No loops longer than
  once-per-page-load.
- Lottie payload sub-budget: **≤ 60KB gzip** (player + all 3 JSONs
  combined). This is *within* the total JS budget in §5.5, not on
  top of it — leaving ≥ 20KB headroom for everything else.
- Animation JSONs hand-authored (3 files); no LottieFiles library
  imports.

### 3.6 Node + package manager + local dev

Bun for both install and run. Matches the agent image which already
uses bun for the channel plugin runtime — same tooling for the bot
dev loop and the site.

Local dev: `cd website && bun run dev` (Vite-backed, fast HMR).

CI runner: stock `oven/bun:1`. No additional CI image work needed.

## 4. Content strategy

### 4.1 Voice

Declarative, terse, terminal-grade. No marketing verbs ("empower",
"unlock", "supercharge"). No exclamation marks. No "we" — the bots
are named agents (`devbot`, `infrabot`); the human is "the operator."
Distance is the brand. If a sentence could appear in a vendor
brochure, rewrite it.

### 4.2 Hero copy

Verbatim from the README (the canonical wording — see top of spec):

- Tagline (display, oversized): **`Your engineering team, running in Kubernetes.`**
- Sub (one line, body mono): **`Persistent, long-lived engineering agents in Kubernetes pods. Real cluster credentials, real PRs, merged to main.`**
- Status strip: **`● running · 2 agents · uptime 14d · PRs shipped this week: 11`**
- CTAs: **`$ read the docs ›`** · **`$ view on github ›`**

(Status strip values are placeholders until the build-time generator
in §3.3 is wired.)

### 4.3 Section lead lines

Drafted now, finalized when implementing:

- "What this is" → mirrors the README opening (see §1.1.2 above) —
  "agent-smith deploys Claude Code as persistent, long-lived
  engineering agents inside Kubernetes pods. Each agent has a
  permanent workspace…" through to "…review comments addressed,
  merged."
- "What your team can do" → 5 bullets verbatim from README (see
  §1.1.3), followed by the Matrix/phone/Claude-desktop sub-line.
- "Under the hood" → leans on existing README content (single-claude
  pod, tmux dance, channel plugin, Stop-hook, NATS); links into
  `/docs/architecture`. Closes with the README's "These agents ship
  work that ends up in `main`."
- "The crew right now" → no prose; just the build-time current
  status block.

### 4.4 Log entry format

Chronological reverse. Each entry one line plus an optional expand
for body/links.

```
2026-05-26T22:14Z  devbot   run=a1b2c3   shipped PR #32 (docs: v1 roadmap)
2026-05-26T19:02Z  infrabot run=d4e5f6   merged PR #28 (chart: setup.command hook)
2026-05-25T11:30Z  devbot   run=g7h8i9   reviewed PR #28 — 8 findings, 4 blocking
```

Frontmatter schema (Astro content collection):

```yaml
---
timestamp: 2026-05-26T22:14:00Z
agent: devbot
run_id: a1b2c3
kind: pr_shipped       # one of: pr_shipped, pr_merged, pr_reviewed,
                       #         incident, release, note
summary: "shipped PR #32 (docs: v1 roadmap)"
link: https://github.com/sherodtaylor/agent-smith/pull/32
---

Optional MDX body for expanded context, multi-paragraph notes, or
embedded blocks.
```

Files: `website/src/content/log/YYYY-MM-DD-<slug>.mdx`.

### 4.5 Log data source

**v1.0 source:** GitHub Actions workflow listening for PR-merge
events in both `sherodtaylor/agent-smith` and `sherodtaylor/homelab`.
Workflow appends an entry file, commits to `main` on a `chore(log):`
commit, site rebuilds.

Manual entries (for non-PR events worth noting) just land as a
hand-written MDX file in the same dir — same schema.

**Deferred to v1.1:** pulling from `#audit` / NATS. PR-merge events
get 80% of the visible surface for ~10 lines of YAML.

### 4.6 Tone in copy

- Glyphs only — no emoji. Allowed: `●` running, `✓` ok, `!` warn,
  `✗` fail.
- Run-ids displayed truncated (first 6 chars) and rendered in
  `--accent`.
- Timestamps in ISO 8601 UTC. No "5 minutes ago" relativization in
  v1 (no JS just for time math).

## 5. Deploy + operations

### 5.1 GitHub Pages config

- Pages source: **GitHub Actions** (not branch-based).
- Deployer: `actions/upload-pages-artifact` + `actions/deploy-pages`.
- Concurrency group: `pages`, cancel-in-progress: `false` (don't
  cancel a deploy mid-flight; just queue).
- Site lives at `https://sherodtaylor.github.io/agent-smith/`.

### 5.2 Custom domain readiness

Zero-work swap when a domain is pointed:

1. Drop `website/public/CNAME` containing the apex/subdomain.
2. Set "Custom domain" in repo Settings → Pages.
3. Flip `base: '/'` and update `site:` in `astro.config.mjs`.
4. Wait for DNS + Pages cert provisioning.

Documented as a 4-line ADR in `website/README.md` so the steps are
recoverable.

### 5.3 404 / error pages

`website/src/pages/404.astro` — terminal-styled `$ ls
/agent-smith/wat` → `wat: No such file or directory` plus a `cd ..`
link back home. No other error pages needed (static site).

### 5.4 robots.txt + sitemap

- Astro's `@astrojs/sitemap` integration generates `/sitemap-index.xml`.
- `website/public/robots.txt` allow-all with a `Sitemap:` line.
- No `noindex` — the site is meant to be found.

### 5.5 Performance budget

Enforced via Lighthouse CI in the same workflow.

| Metric | Budget |
|---|---|
| LCP (4G profile) | < 1.5s |
| First-load JS (gzip) | ≤ 80KB |
| First-load CSS (gzip) | ≤ 20KB |
| Total transferred bytes (landing) | ≤ 250KB |

Hard CI fail if exceeded.

Drivers of the budget: no hero image, fonts self-hosted with `swap`,
Lottie lazy-loaded only on pages that use it, no analytics SDK, no
CSS framework.

### 5.6 Accessibility

WCAG AA minimum:

- Contrast (see §2.1) passes AA for body, AAA for body+accent.
- `prefers-reduced-motion` honored by Lottie and cursor blink (both
  fall back to static frames).
- All ASCII boxes have `role="img"` + `aria-label` describing the
  diagram in prose.
- Skip-to-content link at top.
- Focus rings visible on dark background (1.5px solid `--accent`).
- Heading hierarchy strict (one `<h1>` per page).

### 5.7 Analytics

**None for v1.** No Plausible, no Umami, no GA. Aligns with the
no-marketing voice. Decision documented in `website/README.md`; revisit
if we ever need to know whether adopters actually land.

### 5.8 Privacy

- No cookies.
- No third-party requests.
- No CDN runtime dependency.
- Fonts self-hosted.
- Lottie player vendored locally.
- Site works offline once loaded.

## 6. Out of scope (v1)

Named to prevent scope drift during implementation. Each may be
pulled into a future minor revision if a concrete need surfaces.

- Light-mode variant.
- I18n (English-only).
- Search across the landing/log (Starlight gives docs search; that's
  enough).
- Live NATS / `#audit` integration in `/log` (deferred — PR-merge
  events suffice for v1).
- Per-agent profile pages (agents are namespaced in `/docs/agents/`
  text; no dedicated landing page each).
- RSS feed for `/log` (cheap to add later; not committed for v1).
- Newsletter / email capture. Ever.
- Comments. Use GitHub issues.

## 7. Open implementation questions

Surface to address during the writing-plans phase, not blocking spec
approval.

1. **Berkeley Mono licensing.** If Sherod has a license, display
   headlines use it; otherwise JetBrains Mono weight 800 is the
   fallback. Confirm before plan-writing.
2. **Crew status data shape.** `website/src/data/crew-status.json` is
   regenerated each build. What fields beyond `{name, role, last_pr,
   last_seen}` does the live block want? Open question; small.
3. **PR-merge → log entry workflow** lives in this repo, but it
   listens for events from both `agent-smith` and `homelab`. Easiest
   path: a `repository_dispatch` from `homelab` into `agent-smith`
   when a PR merges. Alternative: cron-poll the gh API. Pick during
   plan.
4. **Lottie JSON authoring tooling.** Hand-roll with `lottie-builder`
   libs, or use After Effects + Bodymovin if Sherod has the licence?
   Affects who can edit them later.
